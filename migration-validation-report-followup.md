### 6. `realtime-dev.supabase-realtime` — harmless, suspicious, or needs investigation?

Inspected (read-only): `ExitCode=0`, `Error=` (empty), clean shutdown logged as `SIGTERM received -
shutting down` at `2026-07-08T10:34:08Z`, `RestartCount=0`, `RestartPolicy=unless-stopped`,
`Health=unhealthy` (stale, from before shutdown — expected once a container is stopped).

**Assessment: most likely a harmless, intentional stop — not a crash — but worth a one-line human
confirmation.** The combination of a clean exit code, no error, and *no* auto-restart despite an
`unless-stopped` policy is the exact signature of an explicit `docker stop` (or a partial
`docker compose stop realtime`): that policy deliberately will not restart a container after a
manual stop until someone runs `docker start`/`compose up` again. There's no repeated-restart or
error pattern in the logs that would suggest a crash loop.

Read-only commands already run: `docker inspect --format ...`, `docker logs --tail 60 --timestamps`.
Additional read-only commands you can run yourself:
```
docker inspect realtime-dev.supabase-realtime --format '{{json .State}}'
docker events --since '2026-07-08T10:00:00' --until '2026-07-08T11:00:00' \
  --filter container=realtime-dev.supabase-realtime   # confirms operator `stop` vs. daemon-issued
```
Do not start it — just confirm intent before treating current state as the migration baseline.
